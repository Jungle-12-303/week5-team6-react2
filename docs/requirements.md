# 요구사항 정의서

## 프로젝트 목표

React의 핵심 기능인 **Component · State · Hooks**를 직접 구현하고,
이를 활용하여 동작하는 웹 페이지를 만든다.

---

## 구현 대상

### 1. Component (컴포넌트)

#### 개념
- UI를 나누는 작은 단위
- 각 Component는 자신만의 로직과 렌더링을 담당
- **반드시 함수형 컴포넌트**로 구현

#### 구현 방식

함수형 컴포넌트를 감싸는 `FunctionComponent` 클래스를 직접 구현한다.

```
FunctionComponent 클래스
  ├── hooks[]       상태 저장 배열
  ├── mount()       최초 렌더링
  └── update()      상태 변경 후 재렌더링
```

#### 제약 조건

| 조건 | 설명 |
|---|---|
| Hook 사용 위치 | 최상위 컴포넌트에서만 사용 가능 |
| State 관리 위치 | 최상위 컴포넌트(루트 컴포넌트)에서만 관리 |
| 자식 컴포넌트 | state를 가지지 않고 부모로부터 전달받은 props만 사용 |
| 자식 컴포넌트 형태 | 상태 없는(Stateless) 순수 함수로 구현 |

> **Lifting State Up 패턴**: 모든 상태는 루트 컴포넌트에서 관리하고,
> 자식 컴포넌트는 props만 받아서 UI만 렌더링한다.

---

### 2. State (상태)

#### 개념
- 값이 변경될 수 있는 동적 데이터
- State가 바뀌면 화면이 자동으로 다시 그려져야 한다

#### 동작 조건
- State 변경 → 해당 컴포넌트 리렌더링 자동 트리거
- 리렌더링 시 이전 상태가 유지되어야 함 (초기값으로 초기화되지 않음)

---

### 3. Hooks

#### 개념
함수형 컴포넌트에서 상태와 생명주기를 사용할 수 있게 해주는 기능

#### 핵심 질문
> "함수는 매번 새로 실행되는데, 상태는 어떻게 유지할까?"
> → 이 문제를 해결하는 것이 Hooks 구현의 핵심이다.

#### 구현 대상

**useState**
- 상태값과 상태 변경 함수를 반환
- 상태 변경 시 컴포넌트 리렌더링 트리거
- 렌더 간 상태 유지

**useEffect**
- 렌더링 이후 실행되는 사이드이펙트 등록
- deps 배열 변경 시에만 재실행
- cleanup 함수 지원

**useMemo**
- deps가 변경될 때만 값을 재계산
- 비용이 큰 연산 결과를 캐싱

---

### 4. Virtual DOM + Diff + Patch

#### 개념
실제 DOM을 직접 조작하는 대신, 메모리 안에서 가상의 DOM 트리를 만들고
변경된 부분만 실제 DOM에 반영한다.

> **새로 직접 구현**한다.

#### 동작 흐름

```
Virtual DOM 생성
  → 이전 Virtual DOM과 비교 (Diff)
  → 바뀐 부분만 실제 DOM에 반영 (Patch)
```

#### 목표
전체를 다시 그리지 않고 **필요한 부분만 업데이트**한다.

---

## 전체 동작 흐름 요약

```
[루트 컴포넌트]
  └── useState/useEffect/useMemo 보유
  └── 자식 컴포넌트에 props 전달

[State 변경]
  → FunctionComponent.update() 호출
  → 컴포넌트 함수 재실행 → 새 Virtual DOM 생성
  → 이전 Virtual DOM과 Diff
  → 변경된 부분만 Patch (실제 DOM 업데이트)

[자식 컴포넌트]
  → props만 받아서 UI 렌더링 (순수 함수)
  → 자체 state 없음
```
