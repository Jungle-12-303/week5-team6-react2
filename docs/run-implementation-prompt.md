# 실행용 구현 프롬프트

## 목적

현재 문서와 코드베이스를 바탕으로, 실제로 실행 가능한 최소 구조부터 구현하고 최종적으로 Vercel 배포 페이지에서 실시간 비트코인 화면이 보이도록 작업을 요청할 때 사용하는 프롬프트입니다.

## 먼저 읽을 문서

- `docs/README.md`
- `docs/requirements.md`
- `docs/core-implementation-prompt.md`
- `docs/bitcoin-ui-prompt.md`
- `docs/realtime-data-connection-prompt.md`
- 필요 시 `docs/01-virtual-dom.md` ~ `docs/06-render-commit-phase.md`

## 실행 목표

- 최종 결과물은 Vercel로 배포된 페이지에서 확인 가능해야 한다.
- 배포된 페이지를 열면 실시간으로 변화하는 비트코인 그래프가 보여야 한다.
- 배포된 페이지를 열면 실시간 계약 히스토리가 보여야 한다.
- 초기 단계에서는 mock 데이터로 시작해도 되지만, 사용자가 페이지를 열었을 때 `실시간으로 변하는 것처럼 보이는 경험`이 반드시 있어야 한다.

## 프롬프트

```text
우리는 React 핵심 기능을 직접 구현하는 프로젝트를 진행 중이고, 현재 기준 브랜치는 develope입니다.

먼저 아래 문서를 반드시 읽고 현재 설계 의도를 반영해 주세요.
- docs/README.md
- docs/requirements.md
- docs/core-implementation-prompt.md
- docs/bitcoin-ui-prompt.md
- docs/realtime-data-connection-prompt.md
- 필요 시 docs/01-virtual-dom.md ~ docs/06-render-commit-phase.md

프로젝트 목표:
- React의 핵심 기능인 Component, State, Hooks를 직접 구현한다.
- Virtual DOM + Diff + Patch도 직접 구현한다.
- 최종적으로는 비트코인 실시간 데이터를 1초 단위로 갱신해서 보여주는 화면을 만든다.
- 참고 UI 예시는 docs/assets/binance-btc-realtime-example.png 및 docs/requirements.md 설명을 따른다.

실행/배포 목표:
- 최종 결과물은 Vercel로 배포된 페이지에서 확인 가능해야 한다.
- 사용자가 배포된 페이지를 열었을 때 실시간으로 변화하는 그래프가 보여야 한다.
- 사용자가 배포된 페이지를 열었을 때 실시간 계약 히스토리가 보여야 한다.
- 초기에는 mock 데이터여도 괜찮지만, 실시간 변화가 눈에 보여야 한다.

중요 제약:
- 반드시 함수형 컴포넌트 기반으로 구현
- 상태는 최상위 루트 컴포넌트에서만 관리
- 자식 컴포넌트는 props만 받는 stateless 순수 함수
- FunctionComponent 클래스 중심 구조 유지
- hooks 배열과 hook index 기반으로 상태 관리
- 상태 변경 시 리렌더링 자동 트리거
- Diff/Patch를 통해 필요한 부분만 DOM 업데이트
- 기존 `react`, `react-dom` 패키지나 React 런타임 사용 금지
- 반드시 우리가 직접 개발한 모듈(패키지/라이브러리)만 사용
- 일반적인 React 구현을 그대로 가져오지 말고 문서 설계를 따라야 함

이번 요청의 목표:
- 현재 코드베이스를 먼저 조사하고, “비트코인 실시간 데이터를 1초 단위로 보여주는 화면”으로 가기 위한 가장 작은 실행 가능한 단계를 실제로 구현해 주세요.
- 코드베이스가 비어 있거나 실행 가능한 최소 구조가 없다면, 문서 설계에 맞는 최소 프로젝트 구조와 엔트리 포인트부터 만들어 주세요.
- 한 번에 전체를 만들지 말고, 지금 단계에서 가장 작은 유효 구현만 진행해 주세요.

우선순위:
1. 현재 코드베이스 조사
2. 실행 가능한 최소 구조가 있는지 판단
3. 없으면 최소 구조 생성
4. 있으면 코어 구현의 다음 단계 진행
5. 가능하면 mock 데이터로 1초 갱신되는 비트코인 화면의 최소 UI 연결
6. 그래프와 계약 히스토리 중 적어도 하나 이상이 실제로 움직이는 상태까지 구현
7. 실행 또는 테스트로 검증
8. 문서와 코드가 어긋나면 관련 문서도 수정

작업 방식:
- 먼저 현재 상태를 짧게 요약
- 그 다음 바로 코드 수정
- 마지막에 변경 파일, 검증 결과, 다음 단계만 간단히 정리

중요:
- 구현 전에 관련 파일과 구조를 먼저 조사할 것
- 필요한 경우에만 새 파일을 만들 것
- 문서에 없는 구조를 임의로 도입하지 말 것
- mock 데이터부터 시작해도 괜찮지만, 1초 단위 갱신 흐름은 반드시 확인 가능해야 함
```
